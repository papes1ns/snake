1. Intro to Artificial Intelligence by Example

When people hear "AI" they tend to picture something opaque and complicated. In practice, most AI is pattern recognition. You give a system a set of inputs, it produces an output, and a training process adjusts internal parameters until the outputs become useful.

This post walks through a complete, working example of that process. The subject is the classic game of Snake. The AI technique is neuroevolution: a population of small neural networks that evolve over generations, getting better at the game through selection pressure alone. No labeled training data. No gradient descent. Just evolution in code.

Everything described here is implemented in a single JavaScript codebase. The neural net, the genetic algorithm, the game engine, and the browser-based debugger are all readable, plain JS. My goal is to demystify what a neural network actually does by showing you every moving part.

2. The Game of Snake

Snake is a grid-based game with simple rules and emergent complexity. A snake moves continuously on a 30×30 board. It eats food to grow longer. Colliding with a wall or its own body ends the game.

What makes Snake interesting as an AI problem is that the optimal strategy changes as the snake grows. Early on, the board is wide open. Later, the snake's own body becomes the primary obstacle. An agent that only learned to chase food in a straight line will eventually trap itself.

Game mechanics that shape training

A few game parameters directly influence how the AI learns. These are defined in scripts/snake/constants.js and mirrored in the headless training engine:

  • Speed ramp: The tick interval uses an arcade-style bit-shift formula: base >> (score >> divisor). The game starts at 100ms per tick and doubles speed every 8 points. This means higher scores are harder to achieve because the snake moves faster.
  • Time limit: Each game is capped at 300 seconds of virtual game time. Agents that stall or circle the edges will time out before reaching high scores.
  • Starvation: If the snake goes too many ticks without eating, it dies. The hunger limit scales with body length: 120 + snake.length * 3. This prevents agents from learning to just survive by looping endlessly.

Together, these mechanics apply evolutionary pressure toward agents that actively seek food and do so quickly.

3. Neural Net Implementation

Each agent's "brain" is a feedforward neural network with the architecture [24, 18, 18, 4]: 24 input neurons, two hidden layers of 18 neurons each, and 4 output neurons (one per direction: North, East, South, West). The agent moves in whichever direction has the highest output value.

Vision: what the snake sees

The snake perceives the board through 8-directional raycasting. From the head, rays are cast in all 8 compass directions (N, NE, E, SE, S, SW, W, NW). Each ray reports three values:

  1. 1 / distance to the nearest wall (closer walls produce larger values)
  2. 1 if food is visible along this ray, 0 otherwise
  3. 1 if the snake's own body is visible along this ray, 0 otherwise

That gives 8 rays × 3 signals = 24 inputs. Here's the implementation from src/headless_game.js: 

HeadlessGame.prototype.getVision = function() {
  var head = this.snake[0];
  var inputs = [];

  for (var r = 0; r < RAY_DIRECTIONS.length; r++) {
    var ray = RAY_DIRECTIONS[r];
    var distance  = 0;
    var foodFound = 0;
    var bodyFound = 0;
    var y = head.y;
    var x = head.x;

    while (true) {
      y += ray.dy;
      x += ray.dx;
      distance++;

      if (y < 0 || y >= this.height || x < 0 || x >= this.width) break;

      if (this.board[y][x] === FOOD  && !foodFound) foodFound = 1;
      if (this.board[y][x] === SNAKE && !bodyFound) bodyFound = 1;
    }

    inputs.push(1 / distance);
    inputs.push(foodFound);
    inputs.push(bodyFound);
  }

  return inputs;
};

Forward propagation

The 24-element vision array feeds into the network. Each layer computes a weighted sum of its inputs, adds a bias term, and applies an activation function. Hidden layers use ReLU (max(0, x)). The output layer uses raw linear values so the agent can compare magnitudes directly.

From src/neural_net.js: 

NeuralNet.prototype.forward = function(inputs) {
  var activation = inputs;

  for (var i = 0; i < this.weights.length; i++) {
    var w = this.weights[i];
    var b = this.biases[i];
    var isOutput = (i === this.weights.length - 1);
    var next = [];

    for (var j = 0; j < w.length; j++) {
      var sum = b[j];
      for (var k = 0; k < activation.length; k++) {
        sum += w[j][k] * activation[k];
      }
      next.push(isOutput ? sum : Math.max(0, sum));
    }

    activation = next;
  }

  return activation;
};

The math for a single neuron is: output = activation(sum(w[i] * input[i]) + bias). Multiply every input by its corresponding weight, sum them, add the bias, then apply the activation function. That's it. A neural network is just this operation repeated across layers.

Weight initialization

Weights are initialized using He initialization, which scales random values by sqrt(2 / fan_in). This prevents signals from exploding or vanishing as they pass through layers. Biases start at zero. 

function randomMatrix(rows, cols) {
  var scale = Math.sqrt(2 / cols);
  var m = [];
  for (var r = 0; r < rows; r++) {
    m[r] = [];
    for (var c = 0; c < cols; c++) {
      m[r][c] = (Math.random() * 2 - 1) * scale;
    }
  }
  return m;
}

Parameter count

This network has (24*18 + 18) + (18*18 + 18) + (18*4 + 4) = 850 total parameters (weights + biases). That's small enough to serialize as a flat JSON array and large enough to encode useful Snake strategies.

4. Training: Reward Functions, Weights, and Bias

Training uses a genetic algorithm. There is no backpropagation. Instead, a population of 1,000 neural networks plays the game, and the best performers reproduce to create the next generation.

Architecture overview

    =================

    [Random Population]         1,000 neural nets with random weights
            |
            v
    [Evaluate]                  Each net plays 8 games, fitness is averaged
            |                       (parallelized across N worker processes)
            v
    [Rank by Fitness]           Sort population by fitness score
            |
            v

            |
            v
    [Tournament Selection]      Pick 5 random candidates, keep the fittest
            |
            v
    [Single-Point Crossover]    Combine two parents at a random splice point
            |
            v
    [Gaussian Mutation]         5% chance per weight, scale 0.3
            |
            v
    [Next Generation]           Repeat for N generations
            |
            v
    [Save best.json]            Best network serialized to disk after each gen

The fitness function

The fitness function is the single most important design decision. It defines what "good" means. Getting it wrong will train an agent that optimizes for the wrong thing.

From src/genetic.js: 

GeneticAlgorithm.prototype.fitness = function(score, steps) {
  return score * score * 5000 + steps;
};

The quadratic term (score * score * 5000) makes each additional food consumed dramatically more valuable than the last. An agent that scores 5 gets 125,000 fitness from score alone, while an agent that scores 10 gets 500,000. This strongly incentivizes food-seeking over passive survival.

The linear + steps term is a tiebreaker. Among agents with the same score, the one that survived longer (and therefore had more chances to eat) ranks higher. For score-zero agents in early generations, this term is the only selection signal: survive longer to get more chances.

An earlier version of this function capped score at 10 with Math.min(score, 10), which caused the population to plateau. Removing the cap was the single biggest training improvement.

Tournament selection

To choose a parent, the algorithm picks 5 random individuals from the ranked population and selects the one with the highest fitness. This balances exploitation (the best agents reproduce more) with exploration (weaker agents still have a chance). 

GeneticAlgorithm.prototype.select = function(results) {
  var best = null;
  for (var i = 0; i < this.tournamentSize; i++) {
    var idx = Math.floor(Math.random() * results.length);
    if (!best || results[idx].fitness > best.fitness) {
      best = results[idx];
    }
  }
  return best.net;
};

Crossover

Two parents produce a child by flattening their weights into a single array and splicing at a random point. Everything before the splice comes from parent 1; everything after comes from parent 2. 

GeneticAlgorithm.prototype.crossover = function(parent1, parent2) {
  var child = new NeuralNet(this.layers);
  var w1 = parent1.getWeightsFlat();
  var w2 = parent2.getWeightsFlat();
  var childWeights = [];

  var crossPoint = Math.floor(Math.random() * w1.length);
  for (var i = 0; i < w1.length; i++) {
    childWeights.push(i < crossPoint ? w1[i] : w2[i]);
  }

  child.setWeightsFlat(childWeights);
  return child;
};

Mutation

After crossover, each weight has a 5% chance of being nudged by a Gaussian random value scaled by 0.3. This introduces small variations that let the population explore nearby regions of the weight space. 

GeneticAlgorithm.prototype.mutate = function(net) {
  var weights = net.getWeightsFlat();
  for (var i = 0; i < weights.length; i++) {
    if (Math.random() < this.mutationRate) {
      weights[i] += gaussianRandom() * this.mutationScale;
    }
  }
  net.setWeightsFlat(weights);
};

Parallel evaluation

Evaluating 1,000 networks across 8 games each is the training bottleneck. The bin/train CLI distributes the population across worker processes using Node.js child_process.fork(). Each worker receives a batch of serialized networks, evaluates them against the headless game engine, and returns fitness scores. The master process then runs selection, crossover, and mutation. 

// src/worker.js
function fitness(score, steps) {
  return score * score * 5000 + steps;
}

process.on('message', function(msg) {
  if (msg.type === 'evaluate') {
    var batch        = msg.batch;
    var gamesPerEval = msg.gamesPerEval;
    var game         = new HeadlessGame();
    var results      = [];

    for (var i = 0; i < batch.length; i++) {
      var net   = NeuralNet.fromJSON(batch[i]);
      var agent = new Agent(net);
      var totalFitness = 0;
      var bestScore    = 0;

      for (var g = 0; g < gamesPerEval; g++) {
        var result = agent.play(game);
        totalFitness += fitness(result.score, result.steps);
        if (result.score > bestScore) bestScore = result.score;
      }

      results.push({
        id:      batch[i].id,
        fitness: totalFitness / gamesPerEval,
        score:   bestScore
      });
    }

    process.send({ type: 'results', results: results });
  }
});

5. Inference

Once training is complete, the best network is saved as a JSON file containing its layer sizes and a flat array of weights. Inference is the process of loading that file and using the network to make decisions in real time.

The Agent loop

The inference loop is remarkably simple. On every game tick: observe the board, run the vision through the network, pick the direction with the highest output value, and move.

From src/agent.js: 

function Agent(neuralNet) {
  this.net = neuralNet;
}

Agent.prototype.decide = function(game) {
  var vision  = game.getVision();
  var outputs = this.net.forward(vision);

  var bestIdx = 0;
  for (var i = 1; i < outputs.length; i++) {
    if (outputs[i] > outputs[bestIdx]) bestIdx = i;
  }
  return DIRECTIONS[bestIdx];
};

Agent.prototype.play = function(game) {
  game.reset();
  while (game.alive) {
    var dir = this.decide(game);
    game.step(dir);
  }
  return { score: game.score, steps: game.steps };
};

Serialization

The model format is intentionally minimal. toJSON() produces an object with two keys: the layer dimensions and a flat array of every weight and bias in order. fromJSON() reconstructs the weight matrices from that flat array. 

NeuralNet.prototype.toJSON = function() {
  return { layers: this.layers, weights: this.getWeightsFlat() };
};

NeuralNet.fromJSON = function(json) {
  var nn = new NeuralNet(json.layers);
  nn.setWeightsFlat(json.weights);
  return nn;
};

The resulting models/best.json file is around 7KB. It contains the full brain of the trained agent. Load it in Node.js for batch evaluation, or fetch it from a web server for browser inference.

Browser inference

In the browser, the same forward pass runs inside scripts/ai/ai_player.js. The AI player monkey-patches the game's moveOne() method. Before each tick, it extracts vision from the live DOM-based game, runs forward(), picks the direction with the highest output, and calls setDirection(). The game engine doesn't know or care that an AI is driving.

6. Recap

The full pipeline, end to end:

  1. Initialize: Create 1,000 neural networks with random weights (He initialization).
  2. Evaluate: Each network plays 8 games of Snake on a headless engine. Fitness = score² * 5000 + steps.
  3. Select: Tournament selection picks parents from the ranked population.
  4. Reproduce: Single-point crossover combines two parents. Gaussian mutation adds small random perturbations.
  5. Elitism: The top 5% carry over unchanged.
  6. Repeat: Steps 2–5 run for N generations. The best network is saved to models/best.json after each generation.
  7. Inference: Load the saved model. On each tick, run forward(vision) and pick argmax(output).

There is no gradient descent. No backpropagation. No labeled data. The only learning signal is the fitness function and the evolutionary process. Networks that play well reproduce. Networks that don't are replaced. Over hundreds of generations, the population converges on strategies that navigate the board, seek food, and avoid collisions.

7. Browser Demo

The project includes a browser-based debugger for watching the AI think in real time. Run bin/serve and open http://localhost:8080.

The 4-column debug layout

When the AI toggle is enabled, the UI shows four columns side by side:

  1. Game Board: The live 30×30 Snake game.
  2. Vision Inputs: A table showing all 8 rays with their wall distance, food presence, and body presence values, color-coded by magnitude.
  3. Network Layers: A visualization of all 4 layers (24 → 18 → 18 → 4) with each neuron colored by its activation value. Green = positive, red = negative, dark = near zero.
  4. Decision Output: Bar chart of the 4 direction scores, with the chosen direction highlighted. Below that: fitness, hunger, speed, and step count.

Speed control

A slider at the top of the decision column controls game speed:

  • 0 (Paused): The game freezes. A "Step" button advances exactly one tick per click, letting you inspect each decision.
  • 1 (Slow): 3 seconds between ticks. Readable in real time.
  • 2 (Normal): The game's natural tick speed.

The virtual timer is synchronized with the debug speed. In slow mode, the countdown timer advances at the same rate as the game ticks, not wall-clock time. This means you can step through a full 300-second game at your own pace.

8. Tips for Building Your Own Training and Inference Pipelines

Building this project taught me a few things about structuring software for neural network training. The core lesson: isolate your parameters. Every tunable value should be independently adjustable without touching unrelated code. Here are the specifics.

Separate the game engine from the renderer

The browser game uses DOM tables, event listeners, and CSS classes. None of that belongs in a training loop. This project has two game engines: scripts/snake/game.js for the browser and src/headless_game.js for training. The headless engine is a plain JavaScript class with zero DOM dependencies. It runs thousands of games per second.

If you're building something similar, write the headless engine first. Get the rules right in isolation. Then build the renderer on top.

Keep the serialization format simple

The model is a JSON object with two keys: layers (the architecture) and weights (a flat array of numbers). That's it. No framework-specific format. No binary encoding. Any language that can parse JSON can load and run the model. This made it trivial to share the same model between Node.js training and browser inference.

Reward what you actually want

Early in development, the fitness function capped score at 10. The population quickly learned to reach score 10 and then stalled. The fix was removing the cap and making score quadratic. The lesson: if the fitness function has a ceiling, your agents will hit it and stop improving.

Similarly, adding + steps as a survival bonus was essential for bootstrapping. In early generations, no agent scores at all. Without the survival term, all agents have zero fitness and selection is random. The linear term gives the algorithm something to select on from generation one.

Randomize starting conditions

If every game starts with the snake in the same position facing the same direction, the population will overfit to that specific scenario. Randomizing the start position and direction on every game forces agents to develop general strategies rather than memorized sequences.

Add constraints that force efficient behavior

Without a time limit, agents can achieve high survival scores by circling the board indefinitely. Without a starvation mechanic, they can avoid food entirely and still accumulate steps. Both constraints push the population toward the behavior you actually want: actively finding and consuming food.

Use parallel workers to scale evaluation

Evaluation is embarrassingly parallel. Each agent's fitness is independent of every other agent. Distributing the population across N worker processes gives near-linear speedup. On a 16-core machine with 15 workers, training runs roughly 10× faster than single-threaded.

Ionize the key parameters

The most productive debugging sessions came from changing a single parameter and observing the effect. That's only possible if each parameter is isolated. In this project, the key tunable values are:

  • Fitness function: score² * 5000 + steps
  • Mutation rate: 5% of weights per generation
  • Mutation scale: 0.3 (Gaussian standard deviation)
  • Elite count: Top 5% of the population
  • Tournament size: 5 candidates per selection
  • Games per evaluation: 8 games averaged
  • Time limit: 100s per game
  • Hunger limit: 120 + length * 3 ticks
  • Network architecture: [24, 18, 18, 4]

Each of these is a named constant or CLI flag. When training stalls, you change one, run a short experiment, and compare. This is the same principle behind good software design: separate concerns, name things clearly, and make each piece independently testable.

The AI doesn't need to be complicated. The game engine, the neural network, the fitness function, and the evolutionary loop are each under 100 lines. The power comes from the interaction between them, tuned by parameters you can see and change.
